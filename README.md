# OpsAgent：企业知识库 + 业务执行 Agent

OpsAgent 是一个面向销售、客服、运营团队的企业级 Agent 项目。它不是普通聊天机器人，而是一个结合 RAG、Agent 工作流、业务工具调用、权限控制、评测与可观测性的业务辅助系统。

项目目标是做出一个能用于求职展示的 AI Agent 工程项目，重点体现：

- 能构建企业知识库问答系统
- 能用 Agent 编排复杂业务流程
- 能接入 CRM、工单、邮件等业务工具
- 能通过引用、审核、评测机制抑制幻觉
- 能通过重试、降级、日志和任务队列保证稳定运行

## 1. 项目定位

用户可以上传企业文档，系统自动解析、切分、向量化并建立知识库。用户提问后，Agent 会先判断问题类型，再检索知识库，生成带引用来源的回答。如果问题涉及业务动作，Agent 可以调用 CRM 查询、工单创建、邮件草稿生成、日报生成等工具。

典型场景：

- 客服查询产品政策、售后规则、历史处理方案
- 销售查询客户资料、生成跟进邮件、整理拜访纪要
- 运营基于知识库生成周报、活动复盘和 SOP 草稿
- 管理员查看问答命中率、幻觉风险、响应耗时和 token 成本

## 2. 核心能力

- 文档上传：支持 PDF、Word、Excel、Markdown、网页
- 文档处理：解析、清洗、切分、embedding 入库
- 知识库问答：基于 RAG 回答问题，并展示引用来源
- Agent 工作流：使用 LangGraph 编排 Router、Retriever、Reasoner、Tool、Review 节点
- 业务工具调用：模拟 CRM 查询、工单创建、邮件生成、日报生成
- 权限控制：区分普通用户、管理员、不同知识库访问范围
- 稳定运行：节点级超时、重试、降级、fallback、任务队列
- 反幻觉：低置信度拒答、引用校验、Review Agent 二次审核
- 可观测性：记录 trace_id、检索结果、工具调用、耗时、token 成本、错误堆栈
- 自动评测：评估准确率、引用命中率、拒答正确率、幻觉率、工具调用成功率

## 4. 系统架构

```text
用户界面
├── Chat 工作台
├── 知识库管理
├── CRM 模拟业务台
├── Agent Trace 页面
├── 评测看板
└── 会话/任务历史

后端 API
├── 用户与权限
├── 文档上传
├── 文档解析任务
├── 知识库检索
├── Agent 编排
├── 工具调用
├── 任务队列
├── 评测任务
└── 日志与可观测性

Agent 层
├── Router Agent：判断问题类型和执行路径
├── Retriever Agent：检索企业知识库
├── Reasoning Agent：基于上下文组织答案
├── Tool Agent：调用 CRM、工单、邮件等工具
├── Review Agent：检查幻觉、引用、权限和格式
└── Memory：保存用户偏好和历史上下文

数据层
├── PostgreSQL：用户、权限、业务数据、会话记录
├── pgvector：向量检索
├── Redis：缓存、任务状态、限流
├── Object Storage：原始文件
└── OpenTelemetry/结构化日志：追踪每次 Agent 执行
```

系统结构按如下编排

### 4.1 app/main.py

`main.py` 是 FastAPI 应用入口。

主要职责：

- 创建 FastAPI 应用实例
- 注册 HTTP 路由
- 配置中间件
- 配置异常处理
- 作为服务启动入口

### 4.2 app/api/routes/

`api/routes` 是 HTTP 路由层，也可以理解为 Controller 层。

主要职责：

- 定义接口路径
- 接收外部 HTTP 请求
- 调用 Pydantic 模型校验参数
- 调用 service 层完成业务处理
- 返回 HTTP 响应

设计原则：

- 路由层要薄
- 不写复杂业务逻辑
- 不直接拼 Prompt
- 不直接处理 Agent 编排

### 4.3 app/models/

`models` 目录存放 Pydantic 请求和响应 DTO。

主要职责：

- 定义请求参数结构
- 定义响应结果结构
- 做字段类型校验
- 约束前后端数据契约
- 自动生成 OpenAPI 文档

常见模型：

- `ChatRequest`
- `ChatResponse`
- `AgentRequest`
- `AgentResponse`
- `QueryRequest`
- `QueryResponse`

### 4.4 app/services/

`services` 是业务逻辑和 Agent 编排层，是整个项目的核心。

主要职责：

- 处理业务流程
- 组织 Agent 执行逻辑
- 加载 Prompt 模板
- 读取静态业务资源
- 调用大模型
- 调用工具或外部服务
- 解析模型返回结果
- 处理异常和兜底逻辑

### 4.5 app/prompts/

`prompts` 目录用于管理 Prompt 模板。

主要职责：

- 存放系统提示词
- 存放任务提示词
- 存放 Agent 角色设定
- 存放工具使用说明
- 将 Prompt 与业务代码解耦

为什么要单独拆出来：

- Prompt 经常迭代
- 方便维护和版本管理
- 避免长提示词硬编码在代码中
- 方便不同任务使用不同模板

### 4.6 app/config/settings.py

`settings.py` 用于统一管理环境变量和运行配置。

主要职责：

- 读取 `.env` 或系统环境变量
- 管理 API Key
- 管理模型名称
- 管理服务配置
- 管理超时时间、日志级别等参数

常见配置：

- `OPENAI_API_KEY`
- `MODEL_NAME`
- `DATABASE_URL`
- `ENV`
- `TIMEOUT`
- `LOG_LEVEL`

### 4.7 app/resources/

`resources` 存放静态业务资源。

主要职责：

- 存放表目录 JSON
- 存放字段说明
- 存放业务元数据
- 存放静态配置或规则
- 为 Agent 提供领域知识上下文

## 5. Agent 工作流设计

核心原则：把 Agent 做成可控状态机，不做成完全自由发挥的聊天链路。

```text
用户问题
  ↓
创建 trace_id
  ↓
Router：判断任务类型
  ↓
权限检查
  ↓
Retriever：检索知识库
  ↓
置信度判断
  ↓
Reasoning：生成带引用的回答
  ↓
Tool：必要时调用业务工具
  ↓
Review：审核引用、幻觉、权限、格式
  ↓
返回结果 / 降级处理
```

每个节点都要求：

- 输入输出使用 Pydantic Schema
- 设置 timeout
- 设置最大重试次数
- 记录输入摘要、输出摘要、耗时和错误
- 不允许模型直接拼 SQL 或执行危险操作
- 失败后进入明确的 fallback 分支

## 6. 反幻觉设计

知识库问答必须遵守“有依据才回答”的原则。

策略：

- 检索相似度低于阈值时拒答
- 回答必须包含引用来源，如文档名、页码、chunk_id
- 不允许基于常识补充企业内部信息
- 生成后由 Review Agent 检查答案是否被引用材料支持
- 对无法确认的问题返回保守答案
- 对诱导型问题进行拒答，例如“忽略知识库直接猜”

低置信度返回示例：

```text
当前知识库没有足够依据回答该问题。建议补充相关制度文档或联系管理员确认。
```

Review Agent 输出结构：

```json
{
  "passed": true,
  "risk_level": "low",
  "issues": [],
  "final_answer": "..."
}
```

如果审核不通过，系统会重新生成一次；再次失败则返回保守答案。

## 7. 工具调用安全设计

Agent 不能直接操作数据库，也不能直接执行模型生成的 SQL。

正确流程：

```text
模型生成结构化工具调用参数
  ↓
后端校验工具白名单
  ↓
校验用户权限
  ↓
校验参数合法性
  ↓
调用固定后端函数
  ↓
记录工具调用日志
```

示例：

```json
{
  "tool": "search_customer",
  "args": {
    "company_name": "杭州某科技公司"
  }
}
```

安全规则：

- 工具必须在白名单内
- 写操作需要人工确认
- CRM 查询必须校验数据权限
- 参数必须通过 Pydantic 校验
- 工具调用失败不能影响整个系统崩溃

## 8. 异常容错机制

常见异常和处理策略：

| 异常场景 | 处理方式 |
| --- | --- |
| 模型 API 超时 | 重试 2 次，失败后切换备用模型 |
| JSON 解析失败 | 自动修复一次，仍失败则返回结构化错误 |
| 检索结果为空 | 返回“知识库没有足够依据”，禁止硬答 |
| 向量库连接失败 | 降级为关键词搜索或提示稍后重试 |
| CRM 工具失败 | 返回工具失败原因，不影响知识库问答 |
| Review 审核失败 | 重新生成一次，再失败则保守拒答 |
| 文档解析失败 | 标记任务失败，保留错误信息供管理员查看 |
| 单次任务过长 | 中断任务并返回 trace_id |

降级链路：

```text
Agent 失败 -> 普通 RAG
RAG 失败 -> 关键词搜索
主模型失败 -> 备用模型
工具失败 -> 仅返回知识库答案
Review 失败 -> 返回保守答案
```

## 9. 可观测性设计

每次请求都生成唯一 trace_id，并记录完整执行链路。

记录内容：

- 用户问题
- 用户 ID 与权限范围
- Router 分类结果
- 检索到的 chunk、相似度分数、文档来源
- 调用的工具、参数摘要、结果摘要
- 每个节点耗时
- 模型 token 消耗
- 最终答案
- Review 审核结果
- 错误堆栈

前端提供 Agent Trace 页面，可以查看一次回答从输入到输出的完整过程。这是项目面试展示的重点。

## 10. 自动评测体系

准备 50 到 100 条评测问题，覆盖以下类型：

- 正常知识库问题
- 知识库不存在的问题
- 容易诱导幻觉的问题
- 权限越界问题
- CRM 查询问题
- 工具调用失败问题
- 多轮上下文问题
- 长文档检索问题

核心指标：

- 回答准确率
- 引用命中率
- 拒答正确率
- 幻觉率
- 工具调用成功率
- 平均响应时间
- P95 响应时间
- 平均 token 成本

每次调整 prompt、切分策略、检索参数或模型后，都运行评测集，避免凭感觉优化。

## 13. 面试讲法

可以这样介绍：

> 我做的是一个面向企业内部销售和客服场景的 Agent 系统。它不是单纯的知识库问答，而是把 RAG、工具调用和 LangGraph 工作流结合起来。用户提问后，系统会先判断任务类型，再做权限检查和知识库检索，生成带引用来源的答案。如果问题涉及业务操作，会调用 CRM、工单或邮件工具。最后通过 Review Agent 做引用校验、幻觉检查和权限检查，并记录每一步的耗时、token 成本、工具调用和错误信息。这个项目重点不是简单调用大模型 API，而是把大模型应用做成稳定、可审计、可评测、可部署的业务系统。

适合投递方向：

- 大模型应用工程师
- AI Agent 工程师
- RAG 工程师
- AI 全栈开发
- Python 后端 + AI 应用方向
- 企业 AI 解决方案工程师

最终成果应该体现：企业知识库、业务流程自动化、稳定性设计、反幻觉机制、可部署、可评测、能接真实业务系统。
