# OpsAgent 项目理解面经

## 一句话项目介绍

OpsAgent 是一个面向企业销售、客服、运营场景的知识库与业务执行 Agent。它不是单纯的聊天机器人，而是把文档解析、RAG 检索、Agent 状态机、业务工具调用、权限控制、Trace 可观测性和自动化测试组合到一起，用来模拟一个可落地的企业 AI 助手后端系统。

面试中可以这样开场：

> 我做的是一个企业级知识库 + 业务执行 Agent。用户可以上传企业制度、产品政策、客服话术等文档，系统会自动解析、清洗、切分、向量化并入库。用户提问后，系统先做权限和任务路由，再通过混合检索找到证据，必要时调用 CRM 查询、工单创建、邮件草稿等白名单工具，最后生成带引用来源的答案，并记录完整 trace，方便排查每次 Agent 执行过程。

## 项目定位和业务流程

这个项目主要解决三类问题：

1. 企业内部知识问答：客服、销售可以基于企业文档询问政策、规则、流程、话术。
2. 业务动作辅助：Agent 可以根据用户意图调用固定后端工具，例如查询客户、创建工单、生成跟进邮件。
3. 工程化治理：通过权限、引用、拒答、Review、Trace、测试集降低大模型幻觉和不可控风险。

核心业务流程可以概括为：

```text
用户登录/携带上下文
  -> 调用 /rag 或 /agent 接口
  -> 权限校验
  -> 创建 trace_id
  -> 文档检索或 Agent 编排
  -> 混合检索召回证据
  -> rerank / 去重 / MMR / 质量检查
  -> 必要时调用白名单业务工具
  -> 生成答案
  -> Review 校验
  -> 返回答案、引用、置信度、工具结果和 trace 信息
```

## 系统分层

项目采用典型的 Python 后端分层结构，代码主要在 `src/ops_agent` 下。

```text
src/ops_agent
  main.py                 FastAPI 应用入口
  api/routes              HTTP 路由层
  models                  Pydantic / dataclass 数据模型
  config                  配置与检索参数
  services                核心业务服务
  services/retriever      企业 RAG 混合检索模块
  services/document_processing
                          文档解析、清洗、切分、元数据、入库
  prompts                 Prompt 模板管理
  resources               静态业务规则
  web                     原生 HTML/CSS/JS 前端页面
storage                   文档、索引、trace 等本地存储目录
tests                     单元测试和稳定性测试
```

路由层只负责接收请求、做权限校验、组织 service 调用和返回结果；真正的业务逻辑放在 services 中。这样面试时可以强调：Controller 层保持薄，复杂逻辑集中在服务层，便于测试和替换实现。

## FastAPI 启动流程

应用入口是 `src/ops_agent/main.py`。启动时会创建 FastAPI 应用，注册多个路由模块，并挂载静态前端页面。

启动生命周期里会做数据库预检：

```text
runtime_state.status = starting
  -> DatabaseService.validate_startup()
  -> DatabaseService.initialize()
  -> 成功则标记 database ready
  -> 失败则标记 degraded
```

这里的设计重点是：外部服务不是永远强依赖。`OPS_AGENT_REQUIRE_EXTERNAL_SERVICES=false` 时，本地开发可以不连接数据库，系统以降级方式启动；生产环境可以要求数据库、pgvector、账号配置都必须可用。

已注册的主要路由包括：

- `/health`：健康检查
- `/auth`：登录认证
- `/users`：用户管理
- `/conversations`：会话管理
- `/rag`：知识库入库、问答、清空
- `/agent`：Agent 执行
- `/evaluation`：评测
- `/tasks`：任务队列
- `/traces`：执行 trace 查询

## 文档入库流程

文档处理主流程在 `services/document_processing/pipeline.py`。

```text
输入 path
  -> DocumentLoader.iter_files()
  -> DocumentLoader.load()
  -> DocumentCleaner.clean()
  -> ChunkerRouter.chunk()
  -> MetadataBuilder.build()
  -> 按 content_hash 去重
  -> VectorStore.upsert_chunks()
```

每个文件会先被解析成统一结构，再做清洗和切分。切分后的 chunk 会补充 metadata，例如来源、标题、内容 hash、章节信息等，最后写入向量库。

这里可以作为面试亮点：

- 支持多格式文档，不把文档处理绑定在单一 PDF 解析器上。
- 入库时用 content_hash 去重，避免重复写入。
- 同一来源重新入库会软删除旧 chunk，保留审计记录。
- 文档解析失败不会中断整个批次，而是记录到 failed 字段中。

## RAG 检索流程

RAG 主流程由 `EnterpriseRagWorkflow` 和 `HybridRetriever` 组成。

```text
用户问题
  -> classify_question 识别问题类型和风险等级
  -> QueryRewriter 改写查询
  -> VectorRetriever 向量召回
  -> KeywordRetriever 关键词召回
  -> merge_results 合并召回结果
  -> MetadataFilter 元数据过滤
  -> hybrid_rank 综合打分
  -> Reranker 重排
  -> Deduplicator 去重
  -> MMR 选择多样化上下文
  -> QualityChecker 判断是否可回答
  -> ContextCompressor 压缩上下文
  -> 返回 sources、confidence、debug 信息
```

混合检索不是简单 top_k 向量相似度，而是把多种信号融合：

- vector_score：语义相似度
- keyword_score：关键词或 BM25 命中
- metadata_score：文档标题、章节、业务字段相关性
- source_bonus：同时被向量和关键词命中时加分
- rerank_score：重排模型或本地重排结果

面试时可以强调：企业知识库里有大量专有名词、制度编号、客户名称，纯向量检索可能漏掉精确词；纯关键词检索又不理解语义，所以项目采用 hybrid retrieval。

## Agent 执行流程

Agent 核心在 `services/agent_service.py`。它实现的是一个显式 ReAct 状态机，而不是让大模型自由循环。

```text
AgentService.run()
  -> _validate_input()
  -> _run_state_machine()
      -> _analyze()        判断 route
      -> _plan()           生成执行计划
      -> _decide_action()  决定下一步动作
      -> _act_and_observe()
      -> _reflect()        判断是否满足最终回答条件
  -> _generate_answer()
  -> _review()
  -> _record_answer()
```

Agent 支持三类 route：

- `knowledge_qa`：只需要知识库问答
- `tool_call`：主要是业务工具调用
- `hybrid`：既需要知识库证据，也需要业务工具结果

可执行动作包括：

- `retrieve_knowledge`
- `search_customer`
- `draft_followup_email`
- `create_ticket`
- `summarize_customer_visit`
- `final_answer`

设计上的重点是“可控”：

- 每个节点都通过 `_run_node` 统一做 retry 和 trace span。
- 工具调用必须经过 ToolRegistry 白名单。
- 低相关度时拒答，不强行生成。
- 生成后通过 Review 检查答案、引用和工具执行结果。
- 最终输出包含 trace_id、route、answer、citations、confidence、tool_results、review。

## 工具调用设计

工具注册在 `services/tool_service.py`。当前实现的是模拟业务工具：

- 查询客户：`search_customer`
- 生成跟进邮件：`draft_followup_email`
- 创建工单：`create_ticket`
- 总结客户拜访：`summarize_customer_visit`

面试时要说明：Agent 不直接操作数据库，也不执行模型生成的 SQL，而是生成结构化 tool call，再由后端白名单工具执行。这样可以控制权限、参数、审计和失败处理。

正确流程是：

```text
模型或规则决定 tool + args
  -> ToolRegistry 检查工具是否存在
  -> 后端函数校验参数
  -> 执行业务逻辑
  -> 返回 ToolResult
  -> TraceRecorder 记录输入输出摘要
```

## 存储与数据库设计

项目支持两套向量存储：

1. 本地 SQLite：默认适合本地开发，索引文件在 `storage/indexes/rag_index.db`。
2. PostgreSQL + pgvector：适合部署，使用 `knowledge_chunks` 表保存 chunk 和 embedding。

`knowledge_chunks` 核心字段包括：

- `chunk_id`
- `document_id`
- `title`
- `text`
- `start_char`
- `end_char`
- `metadata_json`
- `embedding`
- `deleted`
- `deleted_at`
- `ingestion_run_id`

数据库初始化还会创建：

- `users`
- `conversations`
- `conversation_messages`
- `ingestion_logs`

项目还会为知识表创建全文检索索引、文档索引、软删除索引和 metadata source 索引。

## 权限和安全

API 层通过 `current_context` 解析当前用户上下文，再由 `PermissionService` 校验动作权限。典型动作包括：

- `rag.ingest`
- `rag.ask`
- `agent.run`

安全设计可以这样讲：

- 路由层先做权限校验，不让未授权请求进入核心 service。
- Agent 工具调用使用白名单，不允许任意工具执行。
- 写操作通过固定后端函数封装，避免模型直接执行危险操作。
- 知识库低置信度时拒答，避免编造企业内部信息。
- Trace 记录用户、角色、知识范围和执行链路，方便审计。

## 可观测性和容错

Trace 是这个项目的工程亮点之一。每次 Agent 运行都会创建 recorder，记录节点执行、工具调用、LLM prompt、原始输出、parse failure 和最终结果。

节点级容错主要体现在：

- `_run_node` 对节点执行做 retry。
- 数据库启动失败时可进入 degraded 状态。
- 文档入库单文件失败不会影响整个批次。
- 检索低分时返回拒答。
- 工具失败会出现在 ToolResult 和 Review issues 中。
- Review 不通过时返回保守答案。

面试时可以强调：这个项目不是只追求“能答”，而是追求“答得有依据、失败可解释、过程可追踪”。

## 面试讲解模板

面试官让你介绍项目时，可以按这个顺序讲：

1. 背景：企业内部知识多、客服销售需要快速查政策，还要能执行跟进动作。
2. 架构：FastAPI 后端，路由层、服务层、文档处理层、检索层、Agent 层、存储层分离。
3. 入库：多格式文档解析、清洗、切分、补 metadata、embedding、写入 SQLite 或 pgvector。
4. 检索：Query rewrite、向量 + 关键词混合召回、metadata filter、rerank、去重、MMR、质量检查。
5. Agent：显式 ReAct 状态机，route、plan、act、observe、reflect、answer、review。
6. 安全：权限校验、工具白名单、低置信度拒答、引用校验。
7. 工程化：Trace、retry、degraded startup、测试集、配置化。
8. 结果：实现一个可演示、可排查、可扩展的企业 AI Agent 原型。

## 高频面试题与解析

### 1. 这个项目和普通 RAG 问答有什么区别？

普通 RAG 更多是“检索文档 -> 拼 prompt -> 调模型”。这个项目在 RAG 外面加了工程化治理和 Agent 编排：有权限校验、混合检索、质量判断、低置信度拒答、工具调用、Review、Trace 和测试。它更像一个企业业务系统，而不是一个 demo chatbot。

### 2. 为什么要做混合检索？

企业文档里有很多专有名词、客户名称、制度编号和固定短语。纯向量检索擅长语义相似，但可能漏掉精确关键词；纯关键词检索能命中精确词，但对表达变化不敏感。所以系统同时做向量召回和关键词召回，再通过综合打分、rerank、去重和 MMR 选最终上下文。

### 3. 为什么需要 rerank？

初次召回的 top_k 主要保证召回率，可能包含噪声。rerank 的目标是提高排序精度，把更能回答问题的 chunk 放到前面。这样可以减少无关上下文进入 prompt，提高答案质量，也能降低 hallucination 风险。

### 4. MMR 的作用是什么？

MMR 用来平衡相关性和多样性。如果只按分数排序，最终上下文可能来自同一个段落或重复内容。MMR 会在保持相关性的同时减少重复，让模型看到更全面的证据。

### 5. 低置信度为什么要拒答？

企业知识库问答不能靠常识补全内部政策。如果检索分数低，说明当前知识库没有足够依据，强答会产生幻觉和业务风险。所以系统通过 `min_relevance_score` 做拒答，提示补充文档或联系管理员。

### 6. Agent 为什么不用完全自由的多轮工具调用？

完全自由的 Agent 难以控制执行路径，也不方便审计。项目使用显式状态机，把流程拆成 analyze、plan、decide_action、act_observe、reflect、answer、review。每一步都有 trace、retry 和明确的输入输出，更适合企业级场景。

### 7. 工具调用怎么保证安全？

Agent 不能直接执行 SQL 或任意代码。它只能生成结构化的工具名和参数，然后由后端 ToolRegistry 检查白名单，再调用固定函数。这样可以统一做参数校验、权限控制、错误处理和日志记录。

### 8. PostgreSQL + pgvector 和本地 SQLite 怎么取舍？

SQLite 适合本地开发和演示，部署简单，不依赖外部服务。PostgreSQL + pgvector 适合生产，能统一管理用户、会话、知识 chunk、向量索引和审计日志，也更容易扩展到多用户和更大数据量。

### 9. 文档重复入库怎么处理？

文档处理时会根据 chunk 内容计算 `content_hash`，批量入库前会去重。同一来源重新入库时，旧 chunk 会被软删除，新 chunk 写入新的 ingestion run，并记录 ingestion log。这样既避免重复召回，也保留了审计信息。

### 10. Trace 记录了什么，为什么重要？

Trace 会记录用户上下文、路由结果、检索结果、工具调用、LLM prompt、原始输出、节点耗时、错误和最终结果。它解决的是 AI 系统“黑盒难排查”的问题。线上出现回答不准、工具失败、延迟过高时，可以通过 trace 复盘链路。

### 11. Review Agent 的作用是什么？

Review 是生成后的安全闸门。它检查答案是否为空、是否缺少引用、工具是否失败等。如果不通过，就返回保守答案，而不是把未验证结果直接给用户。它的目标是降低幻觉、格式错误和工具失败带来的风险。

### 12. 如果模型 API 超时怎么办？

项目在节点执行层有 retry 机制。对于 LLM 调用，失败后会记录 trace，并可以回退到本地组合答案或保守答案。整体思路是：局部失败不能让整个系统不可解释地崩掉，要么降级，要么明确返回失败原因。

### 13. 权限控制放在哪里？

权限控制放在 API 路由入口。路由通过请求 header 或 session 构造用户上下文，再由 PermissionService 校验具体动作。这样未授权请求不会进入 RAG 或 Agent 核心流程。

### 14. 这个项目还有哪些可以优化的地方？

可以继续优化的点包括：

- 引入真正的 LangGraph StateGraph，把显式节点映射成图。
- 接入真实 CRM、工单和邮件系统。
- 增加更严格的工具参数 Pydantic schema。
- 引入 OpenTelemetry，把 trace 接入外部观测平台。
- 构建标准评测集，持续比较 prompt、chunk、rerank 参数变化。
- 增加多租户知识库隔离和细粒度文档权限。

### 15. 面试官问“你在项目中最核心的贡献是什么”怎么答？

可以这样回答：

> 我核心做的是把一个普通知识库问答系统工程化成可控的 Agent 系统。具体包括文档入库 pipeline、混合检索链路、ReAct 状态机、工具白名单、低置信度拒答、Review 校验和 Trace 记录。重点不是单次调用模型，而是把模型、检索、工具和权限串成一个可测试、可排查、可扩展的后端服务。

## 项目亮点总结

- 不是简单聊天机器人，而是企业知识库 + 业务工具执行 Agent。
- 文档处理链路完整，支持多格式解析、清洗、切分、metadata 和去重。
- 检索链路采用 hybrid retrieval，提高企业专有词和语义问题的命中率。
- Agent 使用显式 ReAct 状态机，流程可控、可追踪。
- 工具调用采用白名单，避免模型直接执行危险操作。
- 支持本地 SQLite 和 PostgreSQL + pgvector 两种向量存储。
- 有权限、拒答、Review、Trace、retry 和 degraded startup 等工程化设计。
- 测试覆盖 RAG、Agent、权限、容错、文档处理和稳定性。
